The standard Lanczos method for computing matrix functions has a brutal memory requirement: storing an $n \times k$ basis matrix that grows with every iteration. For a 500,000-variable problem needing 1,000 iterations, that's 4 GB just for the basis. The textbook solution is the two-pass Lanczos algorithm, which trades doubled computation for constant O(n) memory. The surprising part? When implemented carefully, the two-pass version isn't just memory-efficient—it can be faster. This post explains why the theory missed something important about memory hierarchy.

* All code is available on GitHub: [two-pass-lanczos](https://github.com/lukefleed/two-pass-lanczos)
* The full technical report with proofs and additional experiments: [report.pdf](/two-pass-lanczos/tex/report.pdf)

# Computing Matrix Functions

The problem is computing $\mathbf{x} = f(\mathbf{A})\mathbf{b}$ where $\mathbf{A}$ is a large sparse Hermitian matrix and $f$ is a matrix function. This appears everywhere: solving linear systems corresponds to $f(z) = z^{-1}$, exponential integrators for PDEs use $f(z) = \exp(tz)$, and many other problems require functions like $f(z) = z^{-1/2}$ or $f(z) = \text{sign}(z)$.

Any function $f$ defined on the spectrum of $\mathbf{A}$ has $f(\mathbf{A}) = p(\mathbf{A})$ for some polynomial $p$. This polynomial interpolates $f$ at the eigenvalues in the Hermite sense. The goal is to compute the action $f(\mathbf{A})\mathbf{b}$ without forming $f(\mathbf{A})$ explicitly.

For dense matrices, the Schur decomposition provides a path at cost $O(n^3)$. For sparse matrices with millions of variables, this is impossible. The solution is projection onto a small Krylov subspace.

## Krylov Projection

The polynomial structure suggests building the solution from vectors generated by repeatedly applying $\mathbf{A}$ to $\mathbf{b}$:

$$\mathcal{K}_k(\mathbf{A}, \mathbf{b}) = \text{span}\\{\mathbf{b}, \mathbf{Ab}, \mathbf{A}^2\mathbf{b}, \ldots, \mathbf{A}^{k-1}\mathbf{b}\\}$$

An orthonormal basis $\mathbf{V}_k = [\mathbf{v}_1, \ldots, \mathbf{v}_k]$ for this subspace allows projecting the operator: $\mathbf{H}_k = \mathbf{V}_k^H \mathbf{A} \mathbf{V}_k$. The Arnoldi process builds this basis iteratively, producing an upper Hessenberg matrix $\mathbf{H}_k$ of dimension $k \times k$. The approximation is:

$$\mathbf{x}_k = \|\mathbf{b}\|_2 \mathbf{V}_k f(\mathbf{H}_k) \mathbf{e}_1$$

This reduces an n-dimensional problem to computing a function of a $k \times k$ matrix, which is cheap when $k \ll n$.

For Hermitian matrices, the projection $\mathbf{H}_k$ must also be Hermitian. A matrix that's both upper Hessenberg and Hermitian is necessarily real, symmetric, and tridiagonal. This gives the symmetric Lanczos process.

# The Lanczos Three-Term Recurrence

The Lanczos process builds the orthonormal basis through a three-term recurrence. Starting from $\mathbf{v}_1 = \mathbf{b}/\|\mathbf{b}\|_2$, each iteration computes:

$$
\beta_j \mathbf{v}_{j+1} = \mathbf{A}\mathbf{v}_j - \alpha_j \mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1}
$$

The scalars $\alpha_j = \mathbf{v}_j^H \mathbf{A} \mathbf{v}_j$ and $\beta_j = \|\mathbf{A}\mathbf{v}_j - \alpha_j \mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1}\|_2$ define a symmetric tridiagonal matrix:

$$\mathbf{T}_k =
\begin{pmatrix}
\alpha_1 & \beta_1 & & \\
\beta_1 & \alpha_2 & \ddots & \\
& \ddots & \ddots & \beta_{k-1} \\
& & \beta_{k-1} & \alpha_k
\end{pmatrix}
$$

The relation $\mathbf{A}\mathbf{V}_k = \mathbf{V}_k \mathbf{T}_k + \beta_k \mathbf{v}_{k+1} \mathbf{e}_k^T$ shows that $\mathbf{T}_k = \mathbf{V}_k^H \mathbf{A} \mathbf{V}_k$ is the projection. The approximation becomes:

$$\mathbf{x}_k = \mathbf{V}_k f(\mathbf{T}_k) \mathbf{e}_1 \|\mathbf{b}\|_2$$

This is simpler than the general Arnoldi case because the three-term recurrence only needs the two most recent basis vectors at each step, not all previous ones.

## The Memory Problem

The computation splits into three phases: generate the basis $\mathbf{V}_k$, solve the small problem $\mathbf{y}_k = f(\mathbf{T}_k) \mathbf{e}_1 \|\mathbf{b}\|_2$ on the tridiagonal matrix, then reconstruct $\mathbf{x}_k = \mathbf{V}_k \mathbf{y}_k$. The first phase runs the three-term recurrence for $k$ steps. The second phase is cheap—$\mathbf{T}_k$ is only $k \times k$. The third phase is where the memory bottleneck appears: the full $n \times k$ matrix $\mathbf{V}_k$ is needed to form the linear combination.

The standard implementation stores all $k$ basis vectors as they're generated, giving memory complexity $O(nk)$. Each iteration adds another n-dimensional vector to storage. For problems where $n$ is in the millions and hundreds or thousands of iterations are needed, this becomes prohibitive.

# The Two-Pass Algorithm

The key insight is that the three-term recurrence is deterministic. Given the same starting vector $\mathbf{b}$ and the stored scalars $\\{\alpha_j, \beta_j\\}$, the exact same sequence of basis vectors $\\{\mathbf{v}_j\\}$ can be regenerated.

In the first pass, the Lanczos recurrence runs for $k$ steps, but only the scalars $\alpha_j$ and $\beta_j$ are stored—not the basis vectors. Each $\mathbf{v}_j$ is discarded after using it to compute $\mathbf{v}_{j+1}$. This gives the tridiagonal matrix $\mathbf{T}_k$ at memory cost $O(k)$ instead of $O(nk)$. The small problem $\mathbf{y}_k = f(\mathbf{T}_k) \mathbf{e}_1 \|\mathbf{b}\|_2$ yields the coefficients.

In the second pass, the basis vectors are regenerated one at a time using the stored scalars. Initialize with $\mathbf{v}_1 = \mathbf{b}/\|\mathbf{b}\|_2$ and apply the recurrence:

$$\mathbf{v}_{j+1} = \frac{1}{\beta_j}(\mathbf{A}\mathbf{v}_j - \alpha_j \mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1})$$

As each $\mathbf{v}_j$ is regenerated, its contribution to the solution accumulates immediately: $\mathbf{x}_k \gets \mathbf{x}_k + (\mathbf{y}_k)_j \mathbf{v}_j$. After $k$ iterations, the full solution $\mathbf{x}_k = \sum_{j=1}^k (\mathbf{y}_k)_j \mathbf{v}_j = \mathbf{V}_k \mathbf{y}_k$ is complete, but the full basis was never stored.

The memory footprint is now $O(n)$: space for the current solution vector $\mathbf{x}_k$, the two most recent basis vectors $\mathbf{v}_j$ and $\mathbf{v}_{j-1}$, and a work vector for the matrix-vector product. The scalar arrays $\\{\alpha_j, \beta_j\\}$ add only $O(k)$ overhead, which is negligible when $k \ll n$.

## The Trade-Off

The cost is doubled matrix-vector products: the first pass requires $k$ applications of $\mathbf{A}$, and the second pass requires another $k$. For sparse matrices, the matrix-vector product $\mathbf{Av}$ is the dominant operation, so the naive complexity model predicts the two-pass method should take roughly twice as long as the standard method.

This is the textbook analysis. Memory is O(n) instead of O(nk), but the algorithm pays for it with 2k matrix-vector products instead of k. The question is whether this trade-off holds in practice.

# Implementation in Rust and Faer

Rust was chosen for this project because the two-pass algorithm's performance depends entirely on controlling memory access patterns. Rust provides explicit ownership semantics and zero-cost abstractions—it's possible to reason about when allocations happen, where data lives, and how it moves through the cache hierarchy.

For the linear algebra primitives, the implementation uses [`faer`](https://github.com/sarah-ek/faer-rs), a modern pure-Rust numerical library. The features that matter for this project:

Stack allocation for temporary storage via `MemStack` that gets pre-allocated once and reused, making the hot path allocation-free.

Matrix-free operators through the `LinOp` trait that defines operators by their action without materializing the matrix.

SIMD-optimized kernels where the `zip!` macro generates vectorized loops that compile to packed SIMD instructions.

Generic over field types—`f32`, `f64`, `c32`, or `c64` through the same code path with zero-cost abstractions.

## Building the Core Recurrence

The Lanczos recurrence step implements the mathematical formula:

$$\beta_j \mathbf{v}_{j+1} = \mathbf{A}\mathbf{v}_j - \alpha_j \mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1}$$

The function signature shows what it needs:

```rust
fn lanczos_recurrence_step<T: ComplexField, O: LinOp<T>>(
    operator: &O,
    mut w: MatMut<'_, T>,
    v_curr: MatRef<'_, T>,
    v_prev: MatRef<'_, T>,
    beta_prev: T::Real,
    stack: &mut MemStack,
) -> (T::Real, Option<T::Real>)
```

The generic parameters `T: ComplexField` allow this to work with `f32`, `f64`, or complex numbers. The `LinOp<T>` bound means any operator implementing the trait works—sparse matrices, dense matrices, or custom matrix-free operators. The `MatMut` and `MatRef` types from faer provide views into column-major matrix storage. The `MemStack` provides scratch space for the operator's internal allocations.

The return type `(T::Real, Option<T::Real>)` gives the diagonal element $\alpha_j$ and optionally the off-diagonal $\beta_j$. If $\beta_j$ is numerically zero, `None` signals breakdown.

Start with the operator application:

```rust
// 1. Apply operator: w = A * v_curr
operator.apply(w.rb_mut(), v_curr, Par::Seq, stack);
```

The `rb_mut()` (reborrow mutable) allows passing the view without consuming it. The `Par::Seq` parameter disables parallelism for this call.

Orthogonalize against the previous vector:

```rust
// 2. Orthogonalize against v_{j-1}: w -= β_{j-1} * v_{j-1}
let beta_prev_scaled = T::from_real_impl(&beta_prev);
zip!(w.rb_mut(), v_prev).for_each(|unzip!(w_i, v_prev_i)| {
    *w_i = sub(w_i, &mul(&beta_prev_scaled, v_prev_i));
});
```

The `zip!` macro from faer creates an iterator over multiple matrices simultaneously. The `unzip!` macro destructures the tuple of references. The compiler sees this pattern and generates SIMD instructions—on modern CPUs, this becomes 4 or 8 subtractions per instruction.

Compute the diagonal element:

```rust
// 3. Compute α_j = v_j^H * w
let alpha = T::real_part_impl(&(v_curr.adjoint() * w.rb())[(0, 0)]);
```

At this point, `w` holds $\mathbf{A}\mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1}$. The inner product extracts a real value because $\mathbf{A}$ is Hermitian.

Orthogonalize against the current vector:

```rust
// 4. Orthogonalize against v_j: w -= α_j * v_j
let alpha_scaled = T::from_real_impl(&alpha);
zip!(w.rb_mut(), v_curr).for_each(|unzip!(w_i, v_curr_i)| {
    *w_i = sub(w_i, &mul(&alpha_scaled, v_curr_i));
});
```

Compute the off-diagonal and check for breakdown:

```rust
// 5. Compute β_j = ||w||_2
let beta = w.rb().norm_l2();

// 6. Check for numerical breakdown
let tolerance = T::Real::from_f64_impl(f64::EPSILON * 1000.0);
if beta <= tolerance {
    (alpha, None)  // Breakdown: Krylov subspace is invariant
} else {
    (alpha, Some(beta))
}
```

If $\beta_j$ is numerically zero, the Krylov subspace is invariant under $\mathbf{A}$ and the iteration cannot continue.

Putting it all together, the complete recurrence step:

```rust
fn lanczos_recurrence_step<T: ComplexField, O: LinOp<T>>(
    operator: &O,
    mut w: MatMut<'_, T>,
    v_curr: MatRef<'_, T>,
    v_prev: MatRef<'_, T>,
    beta_prev: T::Real,
    stack: &mut MemStack,
) -> (T::Real, Option<T::Real>) {
    // 1. Apply the operator: w = A * v_curr
    operator.apply(w.rb_mut(), v_curr, Par::Seq, stack);

    // 2. First orthogonalization step (against v_{j-1})
    let beta_prev_scaled = T::from_real_impl(&beta_prev);
    zip!(w.rb_mut(), v_prev).for_each(|unzip!(w_i, v_prev_i)| {
        *w_i = sub(w_i, &mul(&beta_prev_scaled, v_prev_i));
    });

    // 3. Compute the diagonal element α_j = v_j^H * w
    let alpha = T::real_part_impl(&(v_curr.adjoint() * w.rb())[(0, 0)]);

    // 4. Second orthogonalization step (against v_j)
    let alpha_scaled = T::from_real_impl(&alpha);
    zip!(w.rb_mut(), v_curr).for_each(|unzip!(w_i, v_curr_i)| {
        *w_i = sub(w_i, &mul(&alpha_scaled, v_curr_i));
    });

    // 5. Compute the off-diagonal element β_j = ||w||_2
    let beta = w.rb().norm_l2();

    // 6. Check for numerical breakdown
    let tolerance = T::Real::from_f64_impl(f64::EPSILON * 1000.0);
    if beta <= tolerance {
        (alpha, None)
    } else {
        (alpha, Some(beta))
    }
}
```

## The Iterator Pattern

The recurrence gets wrapped in a stateful iterator maintaining the three-term state:

```rust
struct LanczosIteration<'a, T: ComplexField, O: LinOp<T>> {
    operator: &'a O,
    v_prev: Mat<T>,       // v_{j-1}
    v_curr: Mat<T>,       // v_j
    work: Mat<T>,         // workspace for Av_j
    beta_prev: T::Real,   // β_{j-1}
    k: usize,
    max_k: usize,
}
```

The vectors are owned (`Mat<T>` not `MatMut`) so they can be swapped efficiently. The implementation of `next_step` shows the pattern:

```rust
fn next_step(&mut self, stack: &mut MemStack) -> Option<LanczosStep<T>> {
    if self.k >= self.max_k {
        return None;
    }

    let (alpha, beta_option) = lanczos_recurrence_step(
        self.operator,
        self.work.as_mut(),
        self.v_curr.as_ref(),
        self.v_prev.as_ref(),
        T::Real::copy_impl(&self.beta_prev),
        stack,
    );

    self.k += 1;

    match beta_option {
        Some(beta) => {
            // Normalize: v_{j+1} = work / β_j
            let inv_beta = T::from_real_impl(&T::Real::recip_impl(&beta));
            zip!(self.work.as_mut()).for_each(|unzip!(w_i)| {
                *w_i = mul(w_i, &inv_beta);
            });

            // Cycle vectors: v_prev <- v_curr <- work
            core::mem::swap(&mut self.v_prev, &mut self.v_curr);
            core::mem::swap(&mut self.v_curr, &mut self.work);
            self.beta_prev = T::Real::copy_impl(&beta);

            Some(LanczosStep { alpha, beta })
        }
        None => {
            Some(LanczosStep { alpha, beta: T::Real::zero_impl() })
        }
    }
}
```

The `mem::swap` calls are zero-cost—they exchange pointers without copying data. On x86-64, swapping two `Mat<T>` structures (which are essentially fat pointers containing a data pointer, dimensions, and stride) compiles to just three `mov` instructions with no memory allocation. The logical flow is: `v_prev` receives `v_curr`'s data, `v_curr` receives the newly computed vector from `work`, and `work` receives the old `v_prev`'s allocation to be reused in the next iteration.

This cycling pattern keeps a fixed working set of three n-dimensional vectors in play. As the iteration progresses, the same memory locations get reused repeatedly. If these three vectors fit in L3 cache (which they will for problems up to roughly n = 1 million on modern CPUs with 30+ MB L3), every access is a cache hit. This is why the algorithm remains cache-friendly even as k grows—the memory footprint stays constant at O(n), and that small constant is exactly three vectors plus workspace.

The breakdown case (`None` from the recurrence step) indicates the Krylov subspace has become invariant under the operator. This happens when $\beta_j = 0$, meaning $\mathbf{v}_{j+1}$ would be the zero vector. Mathematically, it signals that $\mathcal{K}_j(\mathbf{A}, \mathbf{b})$ contains an exact solution to the problem, so continuing the iteration is pointless. The algorithm terminates early with the current basis.

## First Pass Implementation

The first pass is where memory efficiency begins. Its job is to run $k$ Lanczos iterations and extract only the scalar coefficients $\{\alpha_j, \beta_j\}$ that define $\mathbf{T}_k$. The basis vectors get computed, used, and immediately discarded. Start with the signature:

```rust
pub fn lanczos_pass_one<T: ComplexField>(
    operator: &impl LinOp<T>,
    b: MatRef<'_, T>,
    k: usize,
    stack: &mut MemStack,
) -> Result<LanczosDecomposition<T::Real>, LanczosError>
where
    T::Real: RealField,
```

The function is generic over the field type `T`, allowing it to work with `f32`, `f64`, or complex numbers through the same code path. The `impl LinOp<T>` parameter means any type implementing the linear operator trait works—sparse matrices from `faer`, custom matrix-free operators, or anything that can apply itself to a vector. The `MatRef` borrows the input vector without taking ownership. The `MemStack` provides pre-allocated scratch space that gets reused across iterations, keeping the hot path allocation-free.

The return type wraps a `LanczosDecomposition` struct that holds the minimal data needed for pass two:

```rust
pub struct LanczosDecomposition<R: RealField> {
    pub alphas: Vec<R>,           // Diagonal of T_k
    pub betas: Vec<R>,            // Off-diagonal of T_k
    pub steps_taken: usize,       // Actual iterations (may be < k due to breakdown)
    pub b_norm: R,                // ||b||_2, needed for scaling
}
```

This structure stores only $2k$ real numbers—the diagonal and off-diagonal elements of $\mathbf{T}_k$. Initialize the storage:

```rust
let b_norm = b.norm_l2();
let mut alphas = Vec::with_capacity(k);
let mut betas = Vec::with_capacity(k - 1);
```

The `with_capacity` hint pre-allocates space for exactly $k$ and $k-1$ elements. This is a minor optimization, but in a function that might run hundreds or thousands of times in a larger computation, it avoids incremental reallocations. The reallocation pattern for a growing `Vec` is: start at capacity 4, double when full. Without the hint, storing 1000 elements would trigger 10 reallocations and move-copies. With it, zero reallocations.

Now that we have storage for the scalars, we need to actually run the iterations. The stateful iterator pattern handles this cleanly:

```rust
let mut lanczos_iter = LanczosIteration::new(operator, b, k, b_norm)?;
let mut steps_taken = 0;
```

This allocates the three n-dimensional work vectors internally. After this point, no more allocations happen in the hot path. The iterator owns these vectors, and they get swapped and reused through all $k$ iterations.

With the iterator ready, the main loop can drive the computation forward:

```rust
for i in 0..k {
    if let Some(step) = lanczos_iter.next_step(stack) {
        alphas.push(step.alpha);
        steps_taken += 1;
```

Each call to `next_step` runs one Lanczos recurrence step internally. The returned `LanczosStep` contains $\alpha_j$ and $\beta_j$. The $\alpha$ values get stored immediately—there are always exactly $k$ of them (one per iteration).

Check for breakdown:

```rust
let tolerance = breakdown_tolerance::<T::Real>();
if step.beta <= tolerance {
    break;
}
```

If $\beta_j \leq \epsilon$ (where $\epsilon$ is roughly $10^3 \cdot$ machine epsilon), the Krylov subspace is invariant and iteration stops. The loop breaks, and `steps_taken` holds the actual iteration count. This is why the return type stores `steps_taken` separately—it may be less than the requested $k$.

Store the off-diagonal coefficient:

```rust
if i < k - 1 {
    betas.push(step.beta);
}
} else {
    break;
}
```

The $\beta$ values are off-diagonal, so there are $k-1$ of them. The guard `i < k - 1` prevents storing $\beta_k$, which is never used. If the iterator returns `None` (meaning $k$ iterations completed), the outer loop breaks.

Once the loop completes, all the data we need for the second pass is sitting in these two scalar vectors. We pack it into the decomposition structure:

```rust
Ok(LanczosDecomposition {
    alphas,
    betas,
    steps_taken,
    b_norm,
})
```

This is a cheap move—the `Vec` storage gets transferred into the struct. On x86-64, this compiles to copying three pointers and a length field, then zeroing the source pointers to prevent the destructor from freeing them.

Putting it all together:

```rust
pub fn lanczos_pass_one<T: ComplexField>(
    operator: &impl LinOp<T>,
    b: MatRef<'_, T>,
    k: usize,
    stack: &mut MemStack,
) -> Result<LanczosDecomposition<T::Real>, LanczosError>
where
    T::Real: RealField,
{
    let b_norm = b.norm_l2();
    let mut alphas = Vec::with_capacity(k);
    let mut betas = Vec::with_capacity(k - 1);

    let mut lanczos_iter = LanczosIteration::new(operator, b, k, b_norm)?;

    let mut steps_taken = 0;

    for i in 0..k {
        if let Some(step) = lanczos_iter.next_step(stack) {
            alphas.push(step.alpha);
            steps_taken += 1;

            let tolerance = breakdown_tolerance::<T::Real>();
            if step.beta <= tolerance {
                break;
            }

            if i < k - 1 {
                betas.push(step.beta);
            }
        } else {
            break;
        }
    }

    Ok(LanczosDecomposition {
        alphas,
        betas,
        steps_taken,
        b_norm,
    })
}
```

The critical insight is what this function *doesn't* do: it never stores the $n \times k$ basis matrix. The `LanczosIteration` struct internally maintains three n-dimensional vectors that get swapped and reused. Each iteration generates a new basis vector, extracts the two scalars needed from it, then discards it by overwriting its storage in the next iteration. The memory high-water mark is constant: three vectors plus the two growing scalar arrays, giving $O(n + k)$ total, which simplifies to $O(n)$ when $k \ll n$.

## Second Pass Implementation

The second pass solves a different problem: given the coefficient vector $\mathbf{y}_k$ from solving the projected problem, compute $\mathbf{x}_k = \mathbf{V}_k \mathbf{y}_k = \sum_{j=1}^k (\mathbf{y}_k)_j \mathbf{v}_j$ without storing the full basis matrix. The strategy is to regenerate each basis vector on demand, accumulate its contribution to the solution, then discard it.

Start with the signature:

```rust
pub fn lanczos_pass_two<T: ComplexField>(
    operator: &impl LinOp<T>,
    b: MatRef<'_, T>,
    decomposition: &LanczosDecomposition<T::Real>,
    y_k: MatRef<'_, T>,
    stack: &mut MemStack,
) -> Result<Mat<T>, LanczosError>
where
    T::Real: RealField,
```

The inputs are the same operator and starting vector from pass one, the `LanczosDecomposition` containing the stored scalars, and the coefficient vector $\mathbf{y}_k$ that defines the solution in the Lanczos basis. The contract is to return $\mathbf{x}_k$ without ever allocating the full $n \times k$ matrix.

The regeneration process needs a slightly different approach than the first pass. Where the first pass computed $\alpha_j$ and $\beta_j$ by taking inner products, the second pass already knows these values—they're sitting in the decomposition structure. This means the recurrence step becomes a pure reconstruction operation:

```rust
fn lanczos_reconstruction_step<T: ComplexField, O: LinOp<T>>(
    operator: &O,
    mut w: MatMut<'_, T>,
    v_curr: MatRef<'_, T>,
    v_prev: MatRef<'_, T>,
    alpha_j: T::Real,
    beta_prev: T::Real,
    stack: &mut MemStack,
) {
    // Apply operator
    operator.apply(w.rb_mut(), v_curr, Par::Seq, stack);

    // Orthogonalize using pre-computed coefficients
    let beta_prev_scaled = T::from_real_impl(&beta_prev);
    zip!(w.rb_mut(), v_prev).for_each(|unzip!(w_i, v_prev_i)| {
        *w_i = sub(w_i, &mul(&beta_prev_scaled, v_prev_i));
    });

    let alpha_scaled = T::from_real_impl(&alpha_j);
    zip!(w.rb_mut(), v_curr).for_each(|unzip!(w_i, v_curr_i)| {
        *w_i = sub(w_i, &mul(&alpha_scaled, v_curr_i));
    });
}
```

This looks almost identical to the first pass recurrence, but notice what's missing: the inner product computation for $\alpha_j$ and the norm computation for $\beta_j$. Those are read from storage instead. This makes each regeneration step slightly cheaper than the original generation—two inner products are replaced by two scalar memory reads.

With this specialized recurrence in hand, we can start setting up the working state:

```rust
let mut v_prev = Mat::<T>::zeros(b.nrows(), 1);
let inv_norm = T::from_real_impl(&T::Real::recip_impl(&decomposition.b_norm));
let mut v_curr = b * Scale(inv_norm);
```

The setup is identical to pass one: `v_prev` starts as zero (since $\mathbf{v}_0 = \mathbf{0}$ by convention), and `v_curr` holds the normalized starting vector $\mathbf{v}_1 = \mathbf{b}/\|\mathbf{b}\|_2$. The multiplication by `Scale` avoids allocating a temporary scaled vector—it's a compile-time marker that gets fused into the operation.

The solution vector needs to start accumulating contributions right away:


```rust
let mut x_k = &v_curr * Scale(T::copy_impl(&y_k[(0, 0)]));
```

This is the first term of the sum: $\mathbf{x}_k = (\mathbf{y}_k)_1 \mathbf{v}_1$. The solution vector will accumulate contributions from each regenerated basis vector. Pre-allocating with the first term avoids a separate zero-initialization followed by an addition.

One more piece of infrastructure before the main loop—a workspace vector for intermediate results:

```rust
let mut work = Mat::<T>::zeros(b.nrows(), 1);
```

This vector holds intermediate results during regeneration. It's the last allocation before entering the loop—from here on, everything operates on pre-allocated storage.

Now we're ready for the main reconstruction loop, which regenerates basis vectors one at a time and accumulates their contributions:

```rust
for j in 0..decomposition.steps_taken - 1 {
    let alpha_j = T::Real::copy_impl(&decomposition.alphas[j]);
    let beta_j = T::Real::copy_impl(&decomposition.betas[j]);
    let beta_prev = if j == 0 {
        T::Real::zero_impl()
    } else {
        T::Real::copy_impl(&decomposition.betas[j - 1])
    };
```

Each iteration loads three scalars from the decomposition. The conditional for `beta_prev` handles the first iteration where $j=0$ and $\beta_{-1}$ is defined as zero. This branch is perfectly predicted after the first iteration—the CPU's branch predictor sees the same pattern (false, true, true, true...) and speculatively executes the else path.

With the coefficients loaded, we can regenerate the next basis vector:

```rust
lanczos_reconstruction_step(
    operator,
    work.as_mut(),
    v_curr.as_ref(),
    v_prev.as_ref(),
    alpha_j,
    beta_prev,
    stack,
);
```

After this call, `work` contains the unnormalized vector $\mathbf{w} = \mathbf{A}\mathbf{v}_j - \alpha_j \mathbf{v}_j - \beta_{j-1}\mathbf{v}_{j-1}$. This is exactly what the first pass computed, but now we're using stored coefficients instead of computing them.

The vector needs normalization to become a proper basis element:

```rust
let inv_beta = T::from_real_impl(&T::Real::recip_impl(&beta_j));
zip!(work.as_mut()).for_each(|unzip!(w_i)| {
    *w_i = mul(w_i, &inv_beta);
});
```

The stored $\beta_j$ gives the norm, so we divide by it to normalize. The reciprocal happens once, then the vectorized loop applies it to all $n$ components. On modern x86-64 with AVX2, this processes 4 doubles per instruction.

With the regenerated basis vector in hand, we can accumulate its contribution to the solution:

```rust
let coeff = T::copy_impl(&y_k[(j + 1, 0)]);
zip!(x_k.as_mut(), work.as_ref()).for_each(|unzip!(x_i, v_i)| {
    *x_i = add(x_i, &mul(&coeff, v_i));
});
```

This is the critical operation: $\mathbf{x}_k \gets \mathbf{x}_k + (\mathbf{y}_k)_{j+1} \mathbf{v}_{j+1}$. The coefficient comes from the solution vector for the projected problem. The `zip!` macro generates a fused multiply-add loop that the compiler turns into SIMD instructions. On x86-64 with FMA support, this becomes a single `vfmadd` instruction per 4 doubles.

Finally, we cycle the vectors to prepare for the next iteration:

```rust
core::mem::swap(&mut v_prev, &mut v_curr);
core::mem::swap(&mut v_curr, &mut work);
```

These swaps set up state for the next iteration: `v_prev` becomes the old `v_curr`, `v_curr` becomes the newly regenerated vector from `work`, and `work` receives the old `v_prev`'s storage to be overwritten. Zero copies, just pointer exchanges. The memory locations stay fixed—the names just rotate around them.

After $k-1$ iterations, the loop completes and returns the accumulated solution. The memory footprint throughout this entire function is exactly four n-dimensional vectors: `v_prev`, `v_curr`, `work`, and `x_k`. As $k$ grows the memory usage doesn't change as the same four vectors get reused.

Putting it all together, the complete second pass:

```rust
pub fn lanczos_pass_two<T: ComplexField>(
    operator: &impl LinOp<T>,
    b: MatRef<'_, T>,
    decomposition: &LanczosDecomposition<T::Real>,
    y_k: MatRef<'_, T>,
    stack: &mut MemStack,
) -> Result<Mat<T>, LanczosError>
where
    T::Real: RealField,
{
    // Initialize vectors
    let mut v_prev = Mat::<T>::zeros(b.nrows(), 1);
    let inv_norm = T::from_real_impl(&T::Real::recip_impl(&decomposition.b_norm));
    let mut v_curr = b * Scale(inv_norm);

    // Initialize solution: x_k = y_1 * v_1
    let mut x_k = &v_curr * Scale(T::copy_impl(&y_k[(0, 0)]));

    let mut work = Mat::<T>::zeros(b.nrows(), 1);

    // Main reconstruction loop
    for j in 0..decomposition.steps_taken - 1 {
        let alpha_j = T::Real::copy_impl(&decomposition.alphas[j]);
        let beta_j = T::Real::copy_impl(&decomposition.betas[j]);
        let beta_prev = if j == 0 {
            T::Real::zero_impl()
        } else {
            T::Real::copy_impl(&decomposition.betas[j - 1])
        };

        // Regenerate v_{j+1}
        lanczos_reconstruction_step(
            operator,
            work.as_mut(),
            v_curr.as_ref(),
            v_prev.as_ref(),
            alpha_j,
            beta_prev,
            stack,
        );

        // Normalize using stored β_j
        let inv_beta = T::from_real_impl(&T::Real::recip_impl(&beta_j));
        zip!(work.as_mut()).for_each(|unzip!(w_i)| {
            *w_i = mul(w_i, &inv_beta);
        });

        // Accumulate: x_k += y_{j+1} * v_{j+1}
        let coeff = T::copy_impl(&y_k[(j + 1, 0)]);
        zip!(x_k.as_mut(), work.as_ref()).for_each(|unzip!(x_i, v_i)| {
            *x_i = add(x_i, &mul(&coeff, v_i));
        });

        // Cycle vectors
        core::mem::swap(&mut v_prev, &mut v_curr);
        core::mem::swap(&mut v_curr, &mut work);
    }

    Ok(x_k)
}
```

The accumulation loop is the heart of the algorithm. The same three vectors get touched repeatedly, keeping them hot in cache. This contrasts with the standard method's single dense matrix-vector product that streams through an uncached $n \times k$ matrix.

## The Clean API

The low-level functions get wrapped in a clean interface:

```rust
pub fn lanczos_two_pass<T, O, F>(
    operator: &O,
    b: MatRef<'_, T>,
    k: usize,
    stack: &mut MemStack,
    mut f_tk_solver: F,
) -> Result<Mat<T>, LanczosError>
where
    T: ComplexField,
    T::Real: RealField,
    O: LinOp<T>,
    F: FnMut(&[T::Real], &[T::Real]) -> Result<Mat<T>, anyhow::Error>,
{
    // First pass: compute T_k coefficients
    let decomposition = lanczos_pass_one(operator, b, k, stack)?;

    if decomposition.steps_taken == 0 {
        return Ok(Mat::zeros(b.nrows(), 1));
    }

    // Solve projected problem: y_k' = f(T_k) * e_1
    let y_k_prime = f_tk_solver(&decomposition.alphas, &decomposition.betas)
        .map_err(|e| LanczosError::from(LanczosErrorKind::SolverError(e.to_string())))?;

    // Scale by ||b||
    let y_k = &y_k_prime * Scale(T::from_real_impl(&decomposition.b_norm));

    // Second pass: reconstruct solution
    lanczos_pass_two(operator, b, &decomposition, y_k.as_ref(), stack)
}
```

The solver closure decouples the algorithm from the specific function $f$. Users provide their own implementation—solving a linear system, computing a matrix exponential, or something custom.
